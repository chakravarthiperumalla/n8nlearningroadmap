DevOps Learning Roadmap

Level: Intermediate | Duration: 20 Days | Commitment: 2 Hours/Day

Master the lifecycle of software delivery through Infrastructure as Code, Container Orchestration, and GitOps.

Course Overview

This roadmap transitions you from basic automation to a modern Cloud Native DevOps mindset. You will focus on Terraform, Kubernetes, GitHub Actions, and Observability. Prerequisites: Basic Linux CLI, understanding of Git, and one programming language (Python/Go/Bash).

Module 1: Advanced Infrastructure as Code (Days 1-4)

Day 1: Terraform Modules & Providers

Focus: Structuring Terraform for reusability.

  • Nested modules vs. Root modules.
  • Provider configuration and alias usage for multi-region deployments.
  • Variable validation and local values.
Task: Create a reusable Terraform module for an AWS S3 bucket with versioning and encryption, then call it from a root configuration.

1. What is the benefit of using the 'source' argument in a module block?

A: It allows you to reuse configuration code from local paths or remote registries like GitHub.

2. How do you handle multiple AWS regions in the same Terraform plan?

A: By using provider aliases and passing the provider to the module or resource.

Day 2: Terraform State Management

Focus: Remote backends and state locking.

  • Remote state storage (S3/GCS).
  • State locking with DynamoDB.
  • Handling terraform.tfstate in a team environment.
Task: Configure an S3 backend with DynamoDB locking for your Terraform project.

1. Why should you never commit .tfstate files to Git?

A: They contain sensitive information in plain text and cause conflicts in team settings.

Day 3: Ansible Fundamentals for DevOps

Focus: Configuration management vs Provisioning.

  • Inventory management (Static vs Dynamic).
  • Playbooks, Roles, and Handlers.
  • Idempotency in Ansible.
Task: Write an Ansible playbook to install Nginx and configure a custom HTML page on a local VM or EC2 instance.

1. What does 'idempotency' mean in the context of Ansible?

A: Executing the same operation multiple times has the same effect as a single execution, without changing the state unnecessarily.

Day 4: The IaC Pipeline (TF + Ansible)

Focus: Orchestrating the two tools.

  • Using Terraform to provision infra and Ansible to configure software.
  • Using Terraform's local-exec or remote-exec (and why to avoid them).
Task: Use Terraform to launch an EC2 and automatically trigger an Ansible playbook to configure it upon creation.

1. Which tool is better for 'Day 0' (Infrastructure creation)?

A: Terraform.

Module 2: Containerization & Kubernetes Deep Dive (Days 5-9)

Day 5: Docker Advanced Patterns

Focus: Optimization and Security.

  • Multi-stage builds for smaller images.
  • Distroless images and Alpine-based images.
  • Docker networking (Bridge, Host, Overlay).
Task: Refactor a bulky Python or Node.js Dockerfile into a multi-stage build, reducing size by >50%.

1. What is the primary advantage of a multi-stage build?

A: It keeps the production image small by excluding build-time dependencies.

Day 6: Kubernetes Architecture & Pods

Focus: Control Plane and Data Plane.

  • API Server, Etcd, Scheduler, Controller Manager.
  • Pod lifecycle and Init containers.
Task: Deploy a multi-container pod (Sidecar pattern) where one container logs to a shared volume and another reads it.

1. Which component is the source of truth for the cluster state?

A: Etcd.

Day 7: K8s Deployments & Services

Focus: Scaling and Exposure.

  • Rolling updates and rollbacks.
  • Service types: ClusterIP, NodePort, LoadBalancer.
  • Selectors and Labels.
Task: Deploy an app with 3 replicas. Perform a rolling update to a new version and then roll it back using kubectl.

1. How do you check the rollout history of a deployment?

A: kubectl rollout history deployment/[name]

Day 8: Storage and Config in K8s

Focus: Persistence and Decoupling.

  • PV, PVC, and StorageClasses.
  • ConfigMaps and Secrets (and their security limitations).
Task: Deploy a MySQL pod that uses a Persistent Volume Claim to store data.

1. What happens to a PVC when a Pod is deleted?

A: The PVC remains (unless explicitly deleted), allowing new pods to mount the same data.

Day 9: Helm Package Management

Focus: Templating K8s manifests.

  • Chart structure (values.yaml, templates/).
  • Helm install, upgrade, and list.
Task: Create a custom Helm chart for a simple web application and deploy it using a values file.

1. Where does Helm store the release information by default?

A: In Secrets within the namespace of the release.

Module 3: Continuous Integration & Deployment (Days 10-13)

Day 10: GitHub Actions Workflows

Focus: Event-driven CI.

  • YAML syntax: triggers, jobs, steps.
  • Contexts, secrets, and environment variables.
Task: Create a GitHub Action that runs unit tests on every Pull Request to the main branch.

1. What is the difference between a Step and a Job in GHA?

A: Jobs run in parallel by default on different runners; Steps run sequentially within a job.

Day 11: CI for Containers

Focus: Building and pushing images.

  • GitHub Container Registry (GHCR) or DockerHub integration.
  • Docker build-push-action.
  • Semantic versioning for images.
Task: Build a CI pipeline that builds a Docker image and pushes it to a registry only when a Git tag is created.

1. Why should you avoid using the :latest tag in production?

A: It makes deployments non-deterministic and rollbacks difficult.

Day 12: GitOps with ArgoCD

Focus: Declarative Continuous Delivery.

  • The GitOps philosophy.
  • Connecting ArgoCD to a K8s cluster and a Git repo.
  • Automatic vs Manual synchronization.
Task: Install ArgoCD in your cluster and deploy an application by pointing it to a GitHub repository containing K8s manifests.

1. What is "Configuration Drift"?

A: When the actual state of the cluster differs from the desired state defined in Git.

Day 13: Deployment Strategies

Focus: Reducing downtime and risk.

  • Blue/Green Deployments.
  • Canary Releases using Ingress Nginx or Istio.
Task: Configure a Canary deployment using Nginx Ingress annotations to send 10% of traffic to a "v2" service.

1. Which strategy requires double the infrastructure resources during the release?

A: Blue/Green deployment.

Module 4: Observability & Monitoring (Days 14-16)

Day 14: Prometheus Metrics

Focus: Pull-based monitoring.

  • Prometheus Architecture.
  • PromQL basics.
  • Node Exporter and Kube-state-metrics.
Task: Install Prometheus via Helm and use PromQL to find the CPU usage of a specific pod.

1. Does Prometheus pull or push metrics from applications?

A: It pulls (scrapes) metrics.

Day 15: Grafana Dashboards

Focus: Visualizing data.

  • Connecting Data Sources.
  • Building custom dashboards.
  • Setting up alerts (Slack/Email).
Task: Create a Grafana dashboard showing Pod CPU, Memory, and Disk I/O. Set an alert for high CPU.

1. What is a "DataSource" in Grafana?

A: A connection to a backend storage like Prometheus, Loki, or InfluxDB.

Day 16: Log Management (Loki/ELK)

Focus: Centralized logging.

  • Log shipping (Fluentd/Promtail).
  • Querying logs with LogQL.
Task: Deploy Loki and Promtail. Use Grafana to view live logs from your Kubernetes namespace.

1. Why is centralized logging important in microservices?

A: Because pods are ephemeral; if they die, their local logs are lost.

Module 5: DevSecOps (Days 17-18)

Day 17: Vulnerability Scanning

Focus: Shifting security left.

  • Static Analysis (SAST) and Container scanning.
  • Using Trivy or Snyk.
Task: Integrate a Trivy scan into your GitHub Actions pipeline to fail the build if "High" vulnerabilities are found in the Docker image.

1. When is the best time to scan for vulnerabilities?

A: During the CI process, before the image is pushed to a registry.

Day 18: K8s Network Policies & RBAC

Focus: Hardening the cluster.

  • Least privilege principle in RBAC.
  • Namespace isolation with NetworkPolicies.
Task: Write a NetworkPolicy that only allows traffic to your Database pod from your Web App pod.

1. What does RBAC stand for?

A: Role-Based Access Control.

Module 6: Advanced Cloud Native (Days 19-20)

Day 19: Introduction to Service Mesh

Focus: Microservice communication.

  • Mutual TLS (mTLS).
  • Traffic splitting and Observability at the network layer.
  • Istio vs. Linkerd.
Task: Install Linkerd and "mesh" a sample application to observe traffic latency between services.

1. What is the primary component injected into pods by a service mesh?

A: A sidecar proxy (e.g., Envoy).

Day 20: Full Review & Final Capstone

Focus: Consolidation of all skills.

  • Reviewing the end-to-end flow.
  • Optimizing pipelines for speed.
Task: Review all documentation and prepare for the Capstone Project.

🚀 Final Capstone Project

Scenario: You are tasked with deploying a secure, scalable 3-tier application (Frontend, Backend, DB) with full automation.

  • Infrastructure: Use Terraform to provision a Kubernetes cluster (EKS/GKE or local Kind/Minikube).
  • CI/CD: Set up a GitHub Action to build Docker images and update a Git manifest repo.
  • CD: Use ArgoCD to deploy the application to K8s.
  • Security: Implement Trivy scans and Network Policies.
  • Monitoring: Provide a Grafana dashboard link showing the app's health.
